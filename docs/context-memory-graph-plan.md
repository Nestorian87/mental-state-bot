# Живий контекст і граф памʼяті

Цей документ описує не окремий режим бота, а фоновий pipeline підтримки памʼяті.
Користувач не має запускати його вручну після кожного запису: система оновлює маленький
релевантний фрагмент памʼяті автоматично, а важчі операції виконує рідко.

## Принципи

- Сирі записи, виправлення, фотографії та часові дані залишаються джерелом правди.
- Граф є індексом смислів і гіпотез, а не біографією та не медичним висновком.
- Живий контекст є окремим шаром підтвердженої або спеціально перевіреної довідки.
- Неактуальні дані не стираються одразу: вони втрачають вагу, стають `stale` або `contradicted`.
- AI не отримує весь граф. У кожен prompt потрапляє лише невеликий релевантний підграф.
- Жодних keyword/root-правил для тлумачення стану, причинності або значення подій.
- Важливе оновлення має evidence із конкретного запису, виправлення або підтвердження користувача.

## Шари памʼяті

### Сирий архів

Незмінні або контрольовано редаговані записи користувача, час, джерело, фотографії,
питання бота та виправлення. Він потрібен для експорту, аудиту й переаналізу.

### Живий контекст

Невеликий шар довідки про те, що зараз важливо для життя користувача: люди, місця,
проєкти, рутини, терміни та актуальні обставини. Елемент потрапляє сюди після окремого
питання й перевірки користувача. Це найнадійніша довідка для розуміння неоднозначних назв.

### Граф памʼяті

Ширший шар автоматичних кандидатів і звʼязків між ними. Він може містити повторювані
теми, патерни, активності, емоційні контексти та звʼязки між людьми/проєктами.
Граф не має дублювати весь живий контекст: підтверджені елементи можна повʼязувати з
графом, але не копіювати як окремі незалежні факти.

### Персональний лексикон

Лексикон не є окремою таблицею чи набором хардкодів. Це спеціальні graph-вузли `kind=lexicon`:
дослівний короткий вираз користувача, його обережне контекстне значення та конкретні evidence-записи.
Такі вузли створює той самий capsule-виклик лише коли значення прямо випливає з контексту або виправлення.
Вони не можуть стати `confirmed` лише через AI, повертаються в prompt як умовна довідка і згасають разом
з іншими непідтвердженими вузлами.

### Семантичні капсули

Контекстні тексти для embeddings. Капсула поєднує сирий запис, питання бота, локальне
вікно дня, аналіз, виправлення, живий контекст і релевантний фрагмент графа.

## Pipeline після змістовного запису

1. Завершити звичайний аналіз запису.
2. Зібрати contextual memory capsule.
3. Зробити один дешевий live AI-виклик, який повертає одночасно:
   - текст капсули;
   - обережних кандидатів вузлів;
   - обережних кандидатів звʼязків;
   - evidence і рівні впевненості.
4. Знайти вже відомі вузли за точним label/alias і локальним lookup.
5. Оновити тільки релевантні вузли, звʼязки та evidence.
6. Створити embedding капсули окремим embedding-викликом, якщо embeddings увімкнені.
7. Не запускати глобальну перебудову й не читати весь граф.

Якщо graph lookup або graph update тимчасово падає, запис і embedding не повинні падати
разом із ним. Невдала частина логується і повторно обробляється пізніше.

## Що змінюється автоматично

Після кожної згадки:

- `last_seen_at` оновлюється;
- повторне evidence збільшує підтримку звʼязку;
- суперечливе evidence знижує впевненість або змінює status;
- aliases і summary можуть уточнюватися тільки на основі нового evidence;
- одноразові дрібниці відкидаються AI ще на етапі capsule/graph extraction.
- короткий вираз із контекстним значенням може стати `lexicon`-вузлом, але не універсальним правилом.

Це локальна підтримка графа, а не окремий користувацький режим.

## Щотижнева ревізія

Разом із підсумком тижня або як незалежна фонова задача:

1. Обмежити роботу кількістю вузлів/звʼязків за один запуск.
2. Застосувати recency decay до непідтверджених вузлів і звʼязків.
3. Позначити слабкі та давно невикористані елементи як `stale`.
4. Локально позначити obvious near-duplicate candidates у `possible_duplicates`, не змінюючи статус вузлів.
5. Зберегти `confirmed` довше, але також дозволити їх оновлення при явній суперечності.
6. Знайти потенційні дублікати embeddings-assisted пошуком.
7. Для кожної пари вирішити: merge, alias, separate, ignore або ask.
8. Не виконувати автоматичне ризиковане merge для чутливих тем.
9. Зберегти звіт ревізії та кількість змін.

Невдала ревізія не повинна затримувати тижневий підсумок.

## Вага та відмирання

Пріоритет памʼяті має враховувати:

- свіжість останнього evidence;
- кількість незалежних підтверджень;
- впевненість AI;
- підтвердження або виправлення користувача;
- суперечності;
- попередню корисність під час retrieval;
- тип памʼяті та її status.

Рекомендована поведінка:

- новий кандидат починає з низької ваги;
- повторне підтверджене evidence поступово підсилює його;
- старе невикористане припущення поступово слабшає;
- `stale` залишається доступним для архіву та можливого revival;
- фізичне видалення допускається лише для технічного шуму без evidence або після чіткої політики.

## Retrieval у реальному використанні

Під час аналізу, генерації питання або підсумку система:

1. визначає через AI, які посилання/назви/смисли можуть бути неоднозначними;
2. шукає точні labels/aliases;
3. додає найближчих активних сусідів;
4. враховує embeddings для near-duplicate і ситуаційного пошуку;
5. передає AI малий підграф із джерелами та впевненістю;
6. записує retrieval log: що було знайдено і для якого завдання.

У prompt завжди має бути явно сказано: знайдений граф є довідкою, а не фактом поточного
моменту, доки поточний запис його не підтверджує.

## Вартість

Базовий шлях після запису:

- один live AI-виклик для capsule + graph candidates;
- один embedding-виклик, якщо embeddings увімкнені;
- звичайні SQL lookup/update без додаткового AI.

Не робити окремий graph AI-виклик після кожного запису. Не запускати embeddings для
службових повідомлень, порожніх відповідей, фото без тексту та дрібних команд.

Дорожчий thinking/heavy model використовувати лише для щотижневого merge/review,
і тільки для обмеженої кількості невпевнених пар.

## Підтвердження користувачем

Користувач не має підтверджувати кожен вузол графа. Питання показується лише коли:

- merge може змінити майбутнє розуміння важливої теми;
- є суперечність між сильними evidence;
- чутливе припущення може бути помилковим;
- AI не може відрізнити окремі поняття.

Приклад:

> Маю припущення, що це чутливе переживання пов'язане з ширшою особистою темою.
> Це одна тема чи їх краще тримати окремо?

Варіанти відповіді мають бути контекстними, але також дозволяти вільний текст.

## Візуалізація та debug

Повний граф залишається доступним для архівного перегляду, але робочі режими мають бути:

- активна памʼять;
- вузол і його сусіди;
- памʼять, використана для конкретного питання/підсумку;
- кандидати, що відмирають;
- потенційні дублікати;
- підтверджене окремо від гіпотез.

Це дозволить бачити не лише «що бот памʼятає», а й «чому він це використав».

### Ручне очищення графа

У меню памʼяті є JSON-експорт та контрольований імпорт графа. Це дозволяє віддати файл сильнішій
моделі для ручного прибирання дублікатів, але не змінює сирі записи, фото, підсумки чи AI-аналізи.
Імпорт спершу санітизує структуру, показує кількість вузлів/звʼязків/evidence і лише після явного
підтвердження замінює graph-шар. Нові зовнішні висновки не можуть автоматично ставати `confirmed`;
виняток — вже окремо підтверджені `life_context` вузли з власного експорту.

## Реалізаційні етапи

- [x] Автоматичне локальне оновлення після запису.
- [x] Capsule + graph extraction в одному live AI-виклику.
- [x] Підтягування локального graph context до нової capsule.
- [x] Базовий decay і щотижневе bounded послаблення ваг без AI.
- [x] Cheap local maintenance: decay + possible duplicate candidates без auto-merge.
- [x] Ручна bounded AI-ревізія possible duplicate pairs з обережним alias/stale для high-confidence непідтверджених вузлів.
- [x] Умовна live-ревізія до двох сильних локальних duplicate-пар після нового запису: тільки коли
  цей запис створив або оновив відповідні вузли, без окремого виклику за відсутності кандидатів.

- [x] Автоматична щотижнева bounded AI-ревізія duplicate-пар після cheap maintenance: вона
